# <Product name> — Product Brief

> **You (the human) write this.** Fill in what you know — rough is fine, blanks are fine. This is the raw input to the `initiating-a-project` skill, which will read it, ask you about anything critical that's missing, and turn it into a full PRD (`docs/product/PRD.md`). Copy this file to `docs/product/product-brief.md` before filling it in.

## The idea in a sentence or two
<What is it? What does it do? Say it like you'd say it to a friend.>

## The problem
<What pain does this solve, and for whom? Why does it matter? Why now?>

## Who it's for
<The people who'll use it. Roles, situations, how technical they are. A concrete example user is worth more than a category.>

## Why it's different / why it'll win
<The one thing this must do better than whatever people use today. The reason it deserves to exist. If you're not sure, say so — this is the most important thing to get right, and the skill will push on it.>

## What success looks like
<How will you know it worked? A number if you have one (users, time saved, error rate), or a plain description if you don't.>

## Must-have features (the first version)
<The handful of things the first usable version needs. Don't list everything — list what makes it worth shipping.>

## Explicitly NOT doing (at least not yet)
<Anything you want to consciously leave out, to keep scope sane.>

## Constraints
<Anything fixed: platform (web / mobile / desktop / CLI), a required tech or vendor, a deadline, a budget, compliance/privacy rules, team size/skills.>

## Anything else
<Inspirations, competitors, prior art, links, half-formed thoughts, worries. Dump it here.>
